# Liveness Detection — Prototype

เว็บเดโมตรวจความมีชีวิต (liveness) บนเบราว์เซอร์ **โดยประมวลผลบนอุปกรณ์** — วิดีโอจากกล้องไม่ถูกส่งไปเซิร์ฟเวอร์ของแอปนี้ ใช้ [MediaPipe Face Landmarker](https://ai.google.dev/edge/mediapipe/solutions/vision/face_landmarker) (`@mediapipe/tasks-vision`) ผ่าน WebGPU/WebGL ตามที่รันไทม์รองรับ

## คุณสมบัติหลัก

- **ชุดคำสั่งสุ่ม**: กระพริบตา, ยิ้ม, หันหน้าซ้าย/ขวา — ลำดับสับเปลี่ยนในแต่ละเซสชัน
- **ตรวจกระพริบ**: ใช้ทั้ง Eye Aspect Ratio (EAR) และ Face Blendshapes จากโมเดล เพื่อลดกรณีรูปสองมิติ
- **ตรวจ micro-motion**: วัดการเปลี่ยนแปลงความสว่างบริเวณใบหน้า — ถ้าใบหน้านิ่งเกินเกณฑ์นานเกินไป เซสชันจะล้มเหลว (ช่วยกันหน้าจอนิ่ง/รูปถือ)
- **ทดสอบจากไฟล์วิดีโอ**: เมื่อไม่มีกล้องหรือต้องการทดสอบออฟไลน์

## รายละเอียดทางเทคนิค

### สแตกและรันไทม์

| องค์ประกอบ | รายละเอียด |
|------------|------------|
| **โมเดล** | MediaPipe **Face Landmarker** (`face_landmarker.task`, float16) — โหลดจาก Google Storage; รันผ่าน **Tasks Vision** (`@mediapipe/tasks-vision`) |
| **WASM** | แพ็กเกจเดียวกันจาก CDN (เช่น jsDelivr) สำหรับ `FilesetResolver.forVisionTasks` |
| **การเร่งความเร็ว** | `delegate: 'GPU'` — ใช้ WebGPU/WebGL ตามที่ runtime รองรับ |
| **โหมด** | `runningMode: 'VIDEO'` — ส่ง timestamp ต่อเฟรมเพื่อติดตามใบหน้า |
| **เอาต์พุต** | 478 จุด landmark + **Face Blendshapes** (เปิด `outputFaceBlendshapes`) |
| **ความเชื่อมั่น** | `minFaceDetectionConfidence` / `minFacePresenceConfidence` / `minTrackingConfidence` = **0.5** |
| **เฟรมเวิร์กหน้าเว็บ** | Vite 4 + `@vitejs/plugin-basic-ssl` (HTTPS ใน dev เพื่อ `getUserMedia`) |

### สัญญาณที่ดึงจากโมเดล

- **EAR (Eye Aspect Ratio)** — จากจุด landmark รอบตาซ้าย/ขวา (ชุดดัชนีเดียวกับงาน blink ทั่วไป) ใช้ตรวจจังหวะหลับตาเมื่อค่า EAR ต่ำกว่าเกณฑ์
- **Face Blendshapes** — เช่น `eyeBlinkLeft` / `eyeBlinkRight`, `mouthSmileLeft` / `mouthSmileRight` ใช้ยืนยันว่าโมเดล “เห็น” การหลับตาแบบ 3D และรอยยิ้มตามเกณฑ์คะแนน
- **Yaw (หันหน้า)** — ประมาณจากตำแหน่งจมูกเทียบกึ่งกลางและความกว้างใบหน้าใน landmark; บน UI ใช้ค่า **กลับซ้าย-ขวา** (`mirroredYaw`) ให้สอดคล้องมุมมองกล้องหน้า

### Active liveness (ชาเลนจ์)

ลำดับขั้น **สุ่ม** 4 อย่าง: กระพริบ → ยิ้ม → หันซ้าย → หันขวา (สลับลำดับทุกเซสชัน)

- **กระพริบ**: ต้องครบ **2 ครั้ง**; EAR ต่ำกว่า **~0.21**; ระหว่างรอบหลับตา blendshape กระพริบต้องสูงถึง **~0.12** อย่างน้อยครั้งหนึ่ง (ไม่งั้นนับว่าเป็นคลื่น EAR ลอยๆ ไม่ผ่าน); มีช่วงห่างขั้นต่ำระหว่างครั้งและ timeout ต่อขั้น
- **ยิ้ม**: ค่าเฉลี่ย smile ซ้าย/ขวา **> ~0.34** และค้างถึง **~550 ms**
- **หันหน้า**: yaw เกินเกณฑ์ (**~±0.14** จากกลาง **~±0.08**) และค้าง **~450 ms** ต่อขั้น; มี timeout

นโยบายหน้าใหญ่: ความกว้างใบหน้าในเฟรมต้องไม่เล็กกว่าเกณฑ์สัมพัทธ์ (`MIN_FACE_WIDTH_NORM`) — ลดกรณีภาพเล็ก/ย่อจากรูปจอ

### Micro-motion (ต้านภาพ/จอนิ่ง)

- Crop ใบหน้าจาก bounding box ของ landmark (มี padding) แล้วย่อเป็น **96×96** — คำนวณความสว่างเทา (luma) ต่อพิกเซล
- เปรียบเทียบเฟรมต่อเฟรม: **ค่าเฉลี่ยความต่างสัมบูรณ์ของเทา** ต่อพิกเซล; หากต่ำกว่าเกณฑ์ **~0.9** สะสมเวลารวมเกิน **~3 วินาที** (หลัง warmup **~1.6 วินาที**) จะ fail — มุ่งเป้าหน้าจอนิ่ง / รูปถือนิ่ง / อยู่ไกลเกินไปจนสัญญาณเล็กน้อย

### Passive PAD (เฮียริสติกบน RGB crop) — `src/padLayer.js`

**ไม่ใช่** โมเดล CNN แยก — เป็นสถิติบนภาพ crop ใบหน้า **128×128** แล้วให้คะแนน “คล้าย” การโจมตีหลายแบบ:

| แฟล็ก | แนวคิดหลัก |
|--------|------------|
| **printLike** | Laplacian มีความแปรปรวนต่ำ + local noise ต่ำ → พื้นผิวแบน เช่น **กระดาษ / ภาพพิมพ์** |
| **screenLike** | แถวของภาพสม่ำเสมอผิดธรรมชาติ + สีเอียงไปทางฟ้า (blue bias) → คล้าย **แสง/พิกเซลจากจอ** |
| **videoBlockLike** | ความแปรปรวนของค่าเฉลี่ยบล็อก 8×8 สูง แต่ความแปรปรวนภายในบล็อกต่ำ → คล้าย artifact จากการบีบอัดวิดีโอ |
| **maskSuspect** | กลางใบหน้าเรียบผิดปกติ + อัตราการไล่เฉดที่ขอบเทียบกึ่งกลางสูง → **heuristic สำหรับหน้ากาก/ขอบวัตถุ** (มี false positive/negative สูง) |

Accumulator ต้องเห็นสัญญาณเดียวกันติดกันหลายเฟรมก่อน fail เพื่อลดสะดุ้งจากแสง:

- `printLike` / `screenLike` / `videoBlockLike`: **7 เฟรมติด** (`PAD_CONSECUTIVE_FRAMES`)
- `maskSuspect`: **9 เฟรมติด** (`PAD_CONSECUTIVE_FRAMES + 2`)

## มันป้องกันได้อย่างไร (ภาพรวม)

1. **ชาเลนจ์แบบสุ่ม** — ไม่คาดเดาลำดับจากรอบก่อน; ต้องมีการเคลื่อนไหวและรูปแบบที่โมเดลยอมรับเป็นขั้นๆ
2. **พหุสัญญาณที่กระพริบ** — EAR กับ blendshape ต้องสอดคล้องกัน ลดการใช้รูปสองมิติที่ทำให้ landmark “หลอก” ได้ง่ายกว่าใบหน้า 3D จริง
3. **micro-motion** — หัวใจต้าน “นิ่งเกินไป” เหมือนถ่ายรูปหรือแสดงวิดีโอคู่กับภาพนิ่งบางจังหวะ
4. **PAD แบบพาสซีฟ** — จับลักษณะพื้นผิว/แพตเทิร์นที่พบได้บ่อยในภาพพิมพ์, จอ, หรือวิดีโอบีบอัด

ทั้งหมดรันบนเครื่องผู้ใช้ — ลดความเสี่ยงด้านความเป็นส่วนตัวและต้นทุน backend สำหรับเฟรมดิบ (แต่ logic ยังโปร่งแสงในไคลเอนต์)

## ป้องกัน spoof แบบไหนได้บ้าง (และแบบไหนยังเสี่ยง)

### ช่วยได้ในระดับ prototype

| แบบโจมตี / สถานการณ์ | เหตุผลโดยย่อ |
|----------------------|---------------|
| รูปพิมพ์ / โปสเตอร์ถือนิ่ง | พื้นผิวแบน (PAD) + micro-motion ต่ำ |
| ภาพบนจอ (มือถือ/เมนิเตอร์) นิ่งหรือมีแพตเทิร์นจอ | PAD แบบ screen-like + อาจถูก micro-motion จับ |
| เล่นคลิปวิดีโอใส่กล้อง | blockiness / ลักษณะบีบอัด (PAD) + ถ้าภาพนิ่งช่วงหนึ่ง micro-motion |
| พยายาม “กระพริบ” แค่ใน 2D โดยไม่มีสัญญาณ 3D จากโมเดล | ต้องผ่านเกณฑ์ blendshape ระหว่างหลับตา |
| ใบหน้าเล็กมากในเฟรม (ภาพเล็ก/แอบถ่ายจากภาพ) | fail จาก `MIN_FACE_WIDTH_NORM` |

### ยังป้องกันไม่ได้เต็มที่ / ต้องเลเยอร์เพิ่ม

| สถานการณ์ | เหตุผล |
|-----------|--------|
| **วิดีโอคุณภาพสูงบนจอ + micro-motion พอใช้** | motion ยังมี; PAD อาจไม่พองามเสมอ — โจมตีระดับสูงยังพาไปใกล้ผ่านได้ |
| **Deepfake / face swap เรียลไทม์** | ระบบนี้ไม่ได้ฝึกมาเฉพาะ ตรวจ deepfake; blendshapes อาจถูกจำลองให้สอดคล้อง |
| **หน้ากากซิลิโคน / prosthetic ที่เนียนใน RGB** | เฮียริสติก `maskSuspect` จับได้บางกรณี แต่จำกัดมาก (false positive กับคนผิวเรียบ/แสงแรง) |
| **Replay จากอุปกรณ์อื่นที่จัดฉาก + แสงดี** | ชาเลนจ์ยังเป็นชุดเดียวกัน — หากมีคลิปที่ทำท่าครบและคุณภาพใกล้กล้องสด อาจยังผ่านได้ |
| **โจมตีที่รู้เกณฑ์ในโค้ด** | logic ฝั่ง client ปรับ countermeasure ยาก — การใช้งานจริงควรมี **ชาเลนจ์จากเซิร์ฟเวอร์**, rate limit, รูปแบบสุ่มเพิ่มเติม, และ/หรือโมเดล PAD ฝั่งเซิร์ฟเวอร์ |

สรุป: โปรโตไทป์นี้ผสม **active challenge + สัญญาณ 3D จาก blendshape + การเคลื่อนไหวเชิงพิกเซล + PAD สถิติ** เพื่อกันชั้นต้นได้หลายเคส “ง่ายๆ” แต่**ไม่ใช่** มาตรฐาน eKYC เดี่ยวๆ โดยไม่มี backend และโมเดลเฉพาะทางเพิ่ม

## ข้อกำหนด

- [Node.js](https://nodejs.org/) เวอร์ชันที่รองรับ ES modules (แนะนำ LTS)
- เบราว์เซอร์สมัยใหม่ที่รองรับ `getUserMedia` และ runtime ของ MediaPipe Tasks

## การติดตั้งและรัน

```bash
npm install
npm run dev
```

จากนั้นเปิด **`https://localhost:5173`** (โปรเจกต์ใช้ Vite พร้อม `@vitejs/plugin-basic-ssl` เพื่อให้กล้องทำงานในโหมด secure context)

- **ครั้งแรก** เบราว์เซอร์อาจเตือนเรื่องใบรับรองชั่วคราว — ให้ยอมรับตามขั้นตอนของเบราว์เซอร์
- **กล้อง**: ควรเข้าผ่าน `https://localhost` — การเปิดผ่าน `http://` หรือ IP ในเครือข่ายอาจทำให้กล้องถูกบล็อก
- ถ้าใช้กล้องไม่ได้ ใช้ปุ่ม **ทดสอบจากไฟล์วิดีโอ** แทน

## ข้อจำกัด (สำคัญ)

นี่เป็น **ต้นแบบทางเทคนิค** ไม่ใช่ระบบยืนยันตัวตนระดับ production

- PAD แบบ heuristic และ micro-motion มีทั้ง **false positive** (แสงจ้า/ผิวเรียบ/กล้องแย่) และ **false negative** (โจมตีคุณภาพสูง)
- วิดีโอบนจอหรือ deepfake ระดับสูงอาจหลีกเลี่ยงแนวทางนี้ได้
- การใช้งานจริงควรมี **challenge จากเซิร์ฟเวอร์**, นโยบาย rate limit, โมเดล PAD/anti-spoof ฝั่งเซิร์ฟเวอร์ และการออกแบบ threat model ตามงานจริง
